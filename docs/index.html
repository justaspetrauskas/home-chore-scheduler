Bulk Room Assignment

Endpoint: POST /api/households/{householdId}/rooms/bulk

Description: Create multiple rooms for a household in a single request. If a room name is not provided but a roomTypeId is, the name will be auto-generated (e.g., "Bedroom 1"). Only household members can use this endpoint.

Request Body:

{
	"rooms": [
		{ "name": "Bedroom 1", "roomTypeId": "abc123" },
		{ "roomTypeId": "def456" },
		{ "name": "Living Room" }
	]
}

Response:

{
	"status": "success",
	"data": {
		"rooms": [
			{
				"id": "...",
				"name": "Bedroom 1",
				"householdId": "...",
				"roomTypeId": "abc123",
				"createdById": "...",
				"createdAt": "...",
				"roomType": { "id": "abc123", "label": "Bedroom", ... }
			},
			// ...more rooms
		]
	}
}

Errors:

  • 403: Not a member of this household

Controller: createRoomsBulk in householdController.js

Realtime Layer (Socket.IO)

The API now includes a Socket.IO server for realtime updates.

  • Socket.IO is initialized in src/server.js.
  • The Express app is wrapped with createServer(app).
  • A Socket.IO server instance is created with shared CORS config.
  • The instance is attached to Express with app.set("io", io).
  • Request handlers can access it using req.app.get("io").

Example usage inside a controller:

const io = req.app.get("io");
io.emit("cleaning-event:created", { eventId: event.id });

Use broadcast events for global updates, then move to rooms (for example by household) when you need more targeted realtime delivery.

Client quick start:

npm install socket.io-client

import { io } from "socket.io-client";

const socket = io("http://localhost:5050", {
	withCredentials: true,
});

socket.on("connect", () => {
	console.log("connected", socket.id);
});

socket.on("cleaning-event:created", (payload) => {
	console.log("new event", payload);
});

Controllers

authController.js

Handles user authentication:

  • register: Registers a new user, hashes password, checks for duplicate emails.
  • login: Authenticates user, verifies password, issues JWT token.
  • logout: Ends user session (token removal/invalid).
  • Security: Uses generic error messages, never exposes sensitive info.

userController.js

Manages user profile and settings:

  • getMe: Returns authenticated user's profile, memberships (with household names), and cleaning events for their default household. If no default, uses the newest household.
  • setDefaultHousehold: Allows user to set their default household.
  • CRUD: Update/delete user (self only), strict access control.
  • Cleans: Removes sensitive fields from responses.

householdController.js

Manages household lifecycle:

  • createHousehold: Creates a household, sets creator as ADMIN.
  • getUserHouseholds: Lists all households for user, including members, rooms, and events.
  • getHouseholdById: Gets a household by ID, with members, rooms, and events.
  • createInvitation: Creates an email invitation for a household member using a secure random token (crypto.randomBytes(32).toString("hex")) and stores it in the Invitation table.
  • inviteMember: Adds a user to a household.
  • removeMember: Removes a user from a household.
  • deleteHousehold: Only owner can delete; deletes memberships first to avoid FK errors.
  • setDefaultHouseholdForUser: Lets user set a household as their default.

invitationController.js

Handles invitation token validation:

  • getInvitationByToken: Looks up invitation by token and validates all required conditions: invitation exists, is not used, and has not expired.

roomController.js

Manages rooms within households:

  • CRUD: Create, read, update, delete rooms.
  • Access: Only household members can manage rooms.
  • List: List rooms by household.

choresController.js

Manages chores:

  • CRUD: Create, update, delete, list, get by ID.
  • Validation: Uses Zod schemas for input validation.
  • Permissions: Only creator or admin can modify/delete.

cleaningEventController.js

Manages cleaning events:

  • createCleaningEvent: Creates an event with householdId, name, eventDate, notificationDate, distributionMode, recurrenceRule, notifyParticipants, and status.
  • getCleaningEvents: Lists events for households where the user is a member.
  • updateCleaningEvent: Partially updates event fields (name, dates, distribution, recurrence, notification flag, status) for authorized household members.

Middleware

authMiddleware.js

  • Verifies JWT tokens from headers/cookies.
  • Attaches authenticated user to request.
  • Blocks access if authentication fails.

corsMiddleware.js

  • (If used) Configures CORS for cross-origin requests.

validateRequestMiddleware.js

  • Validates request bodies using Zod schemas.
  • Returns 400 errors for invalid input.

API Routes

/auth

  • POST /register — Register a new user.
  • POST /login — Authenticate and receive a JWT.
  • POST /logout — End session.

/users

  • GET /users — List all users (auth required).
  • GET /users/me — Get current user profile, memberships, and cleaning events.
  • PUT /users/:id — Update user (self only).
  • DELETE /users/:id — Delete user (self only).
  • POST /users/set-default-household — Set default household.

/households

  • POST /households — Create a new household.
  • GET /households — List user’s households (with rooms/events).
  • GET /households/:id — Get household details (with rooms/events).
  • POST /household/:id/invite — Create invitation token for an email and store it in DB.
  • POST /households/:householdId/invite — Invite a user.
  • DELETE /households/:householdId/members/:userId — Remove a member.
  • DELETE /households/:id — Delete a household (owner only).
  • POST /households/:householdId/set-default — Set as user’s default household.

/invite

  • GET /invite/:token — Validate invitation token (exists, not used, not expired).

Invitation Endpoints Details

Create invitation

Endpoint: POST /household/:id/invite

Description: Creates a household invitation for the provided email. The requester must be a member of the household. A secure random token is generated and persisted.

Request Body:

{
	"email": "person@example.com"
}

Success Response (201):

{
	"id": "...",
	"householdId": "...",
	"email": "person@example.com",
	"token": "<64-hex-token>",
	"used": false,
	"createdAt": "...",
	"expiresAt": "..."
}

Errors:

  • 400: Email is required
  • 403: Not a member of this household

Validate invitation token

Endpoint: GET /invite/:token

Description: Validates an invitation token by checking that it exists, has not been used, and has not expired.

Success Response (200):

{
	"status": "success",
	"data": {
		"invitation": {
			"id": "...",
			"householdId": "...",
			"email": "person@example.com",
			"token": "<64-hex-token>",
			"used": false,
			"createdAt": "...",
			"expiresAt": "...",
			"household": {
				"id": "...",
				"name": "My Home"
			}
		}
	}
}

Errors:

  • 404: Invitation not found
  • 400: Invitation already used
  • 400: Invitation expired

/rooms

  • POST /rooms — Create a room.
  • GET /rooms/household/:householdId — List rooms for a household.
  • GET /rooms/:id — Get room details.
  • PUT /rooms/:id — Update a room.
  • DELETE /rooms/:id — Delete a room.

/chores

  • GET /chores — List chores.
  • GET /chores/:id — Get chore details.
  • POST /chores — Create a chore.
  • PUT /chores/:id — Update a chore.
  • DELETE /chores/:id — Delete a chore.

/cleaning-events

  • GET /cleaning-events — List cleaning events for the user.
  • POST /cleaning-events — Create a new cleaning event.
  • PATCH /cleaning-events/:id — Update an existing cleaning event.

Interview-Ready Summary

  • Architecture separates business logic (controllers), validation (middlewares), and routing (routes).
  • Security is enforced at middleware and controller level.
  • Data returned is always cleaned of sensitive fields.
  • Most endpoints are protected by authentication; public invite token validation is exposed at GET /invite/:token by design.
  • The API is RESTful, modular, and follows best practices for error handling and data validation.
